# E-008 — PRD AC-3.2 documenta incorrectamente la restricción de fondos retenidos

## Contenido capturado
```markdown
### 1.3 Wallet & Bid Fund Locking
**Acceptance Criteria:**
- AC-3.1: Balance cannot go negative
- AC-3.2: Held funds cannot exceed balance
- AC-3.3: Every balance change produces a Ledger entry
- AC-3.4: Deposit amount mismatch (user input vs payment) throws `PaymentMismatchException`
```

Y en WalletService (holdFunds), el comportamiento real:
```typescript
const newBalance = currentBalance.minus(amountDecimal); // balance baja
const newHeld = new Decimal(newHeldNum);                // heldFunds sube
```

## Observación
El PRD AC-3.2 dice "Held funds cannot exceed balance" pero el comportamiento real es correcto y necesariamente diferente: cuando un usuario bloquea fondos para una puja, `balance` (fondos disponibles) disminuye y `heldFunds` aumenta. Después del bloqueo, `heldFunds` puede ser mayor que `balance` (ej: si un usuario con 100 MXN puja 80 MXN, termina con balance=20, heldFunds=80, que viola literalmente AC-3.2). La restricción real implementada en código es "no puedes bloquear más de tu balance disponible actual" — que es la restricción correcta de dominio. El PRD documenta mal este invariante.

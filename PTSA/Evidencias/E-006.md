# E-006 — CLIENT pasa apiUrl a templates — browser puede llamar directamente a la API

## Contenido capturado
```typescript
const API_URL = process.env.API_URL || 'http://localhost:3000';
const BASE_URL = process.env.BASE_URL || 'http://localhost:5174';

// En cada handler de página:
return { profile, wallet, bids, auctions, apiUrl: API_URL, baseUrl: BASE_URL };

// Ejemplo de múltiples páginas:
return { profile, apiUrl: API_URL };
return { settings, apiUrl: API_URL };
return { bids, page, apiUrl: API_URL };
return { wallet, apiUrl: API_URL };
return { apiUrl: API_URL };  // páginas sin data server-side (deposit, withdraw)
```

## Observación
El `AppController` del sitio CLIENT pasa `apiUrl: API_URL` a todos los templates Nunjucks. Esto hace que el JavaScript del browser que se ejecuta en las páginas renderizadas por CLIENT tenga acceso al URL del API. El sitio implementa BFF server-side (extrae el JWT del cookie HttpOnly para llamadas SSR), pero también expone la URL del API al browser para llamadas JavaScript directas. En páginas como `/wallet/deposit` y `/wallet/withdraw`, devuelve solo `{ apiUrl: API_URL }` sin data pre-cargada, lo que sugiere que estas páginas hacen todas sus operaciones via JavaScript del browser directamente al API. El mecanismo de autenticación para estas llamadas directas no está verificado en este análisis (requeriría inspección de templates JS del browser).

# H-006 — CLIENT SSR expone API_URL al browser — patrón BFF parcialmente implementado

## Descripción
El `AppController` del sitio CLIENT pasa `apiUrl: API_URL` como variable de template a todas las páginas Nunjucks. Esto hace que el JavaScript del browser que se ejecuta en las páginas renderizadas tenga acceso al URL del API y pueda realizar llamadas directas desde el lado del cliente.

El patrón BFF está correctamente implementado en el servidor (extrae el JWT del cookie HttpOnly mediante `getToken(req)` para llamadas SSR), pero la exposición de `apiUrl` permite que el browser JS también llame directamente al API — lo que es el anti-patrón que BFF precisamente evita.

En páginas como `/wallet/deposit` y `/wallet/withdraw`, el controller devuelve exclusivamente `{ apiUrl: API_URL }` sin datos pre-cargados, indicando que estas páginas dependen completamente de llamadas browser→API directas.

## Evidencia Clave
- **E-006**: `src/apps/client/src/app.controller.ts` líneas 5-6, 39-52 — `apiUrl: API_URL` en todos los handlers

## Impacto en el Dominio
El patrón BFF documentado en la arquitectura (`01-Platform-Overview.md`, `06-Backend-Architecture.md`) establece que "Client-side JavaScript never has direct access to tokens". Si las páginas de wallet hacen llamadas directas al API desde el browser con tokens visibles en localStorage o Authorization headers (en lugar de cookies HttpOnly), se viola el invariante de seguridad de tokens.

**Confidence limitada (80%)**: No se ha leído el JavaScript de browser en `public/js/pages/` para confirmar cómo esas llamadas directas gestionan la autenticación. Si usan cookies HttpOnly (mismo mecanismo), el riesgo es menor. Si usan otro mecanismo, el riesgo es alto.

## Riesgo Calculado
- Impacto: 2 (medio — potencial exposición de tokens si JS de browser no usa cookie auth)
- Probabilidad: 2 (media — condicionada a confirmación del patrón de autenticación en JS browser)
- Risk_bruto: 4

## Recomendación
1. Auditar `src/apps/client/public/js/pages/wallet/` para verificar cómo el JS de browser gestiona la autenticación en llamadas al API.
2. Si usa tokens en localStorage o headers: migrar a fetch con `credentials: 'include'` (cookies HttpOnly) o crear endpoints proxy BFF en el CLIENT para estas operaciones.
3. Si ya usa `credentials: 'include'`: documentar el patrón y reducir severidad a BAJA (el uso de `apiUrl` sería solo para construir la URL de fetch, no un bypass de seguridad).

**PT sugerido**: INVESTIGACIÓN → BUG/REFACTOR dependiendo del hallazgo de la auditoría de JS browser

## Revisión R-001 — 2026-06-23 — PT-028 Investigation completada

**Estado anterior**: ABIERTA MEDIA (confidence 80%)
**Estado nuevo**: ABIERTA **BAJA** (confidence 98%)

**Hallazgo de investigación (PT-028)**:
- No existen archivos JS bajo `src/apps/client/public/js/` — solo CSS
- El JS de browser está inline en los templates Nunjucks (`{% block scripts %}`)
- Ambas páginas de wallet verificadas:
  - `deposit.html` línea 32: `fetch(API + '/api/v1/wallet/deposit', { credentials: 'include', ... })`
  - `withdraw.html` línea 26: `fetch(API + '/api/v1/wallet/withdraw', { credentials: 'include', ... })`
- **H1 confirmada**: `credentials: 'include'` envía automáticamente la cookie HttpOnly con el JWT
- El token JWT NUNCA es accesible al JS del browser (está en HttpOnly cookie)

**Reclasificación**:
- Severidad: MEDIA → **BAJA** (el invariante de seguridad "tokens not accessible to JS" se cumple)
- Penalización: 5 → 1
- Risk_bruto: 4 (2×2) → 2 (1×2)
- El `apiUrl` expuesto en templates es solo para construir la URL del fetch — no es un riesgo de seguridad dado `credentials: 'include'`

**Recomendación actualizada**: BAJA prioridad. El BFF está implementado correctamente a nivel de token. Como mejora futura (no urgente), las páginas de wallet podrían migrar a SSR completo para eliminar la dependencia del browser JS. Sin PT urgente.

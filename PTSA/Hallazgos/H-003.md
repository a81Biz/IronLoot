# H-003 — PaymentsService usa Logger estándar de NestJS en lugar de StructuredLogger

## Descripción
`PaymentsService` instancia `new Logger(PaymentsService.name)` de `@nestjs/common`. Todos los demás servicios del stack de pagos y financiero auditados (`BidsService`, `WalletService`, `AuthService`) usan el `StructuredLogger` del módulo de observabilidad interno (`../../common/observability`), que propaga `traceId` y emite logs en formato JSON estructurado.

El impacto es que los logs del módulo de pagos (webhooks, verificación, depósitos, retiros) no incluyen `traceId`, cortando la cadena de trazabilidad cuando un pago falla o un webhook es rechazado.

## Evidencia Clave
- **E-004**: `src/api/src/modules/payments/payments.service.ts` líneas 1-12 — `import { Logger } from '@nestjs/common'`

## Impacto en el Dominio
Los webhooks de pago son el punto de integración más crítico con Mercado Pago / PayPal / HeyBanco. Cuando un webhook falla (HMAC inválido, pago duplicado, estado inesperado), la capacidad de diagnóstico post-mortem es limitada porque los logs de `PaymentsService` no tienen `traceId`.

El impacto es BAJA porque el sistema igual registra eventos clave (el pago sí llega a `LedgerEntry`), pero la trazabilidad end-to-end browser→API→payment→wallet queda rota en el segmento de pagos.

## Riesgo Calculado
- Impacto: 1 (bajo — no afecta comportamiento funcional, solo observabilidad)
- Probabilidad: 2 (media — la ausencia de traceId en payments se manifestará en el primer incidente que requiera diagnóstico)
- Risk_bruto: 2

## Recomendación
Cambiar el import y la declaración del logger en `PaymentsService` para usar `StructuredLogger` igual que los demás servicios financieros:
```typescript
import { StructuredLogger } from '../../common/observability';
private readonly logger = new StructuredLogger(PaymentsService.name);
```
